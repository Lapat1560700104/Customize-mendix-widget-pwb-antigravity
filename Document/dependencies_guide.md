# คู่มืออธิบายการทำงานและความสำคัญของ Dependencies ในโปรเจค
*(Dependencies, devDependencies, Overrides & Resolutions Reference Guide)*

เอกสารฉบับนี้จัดทำขึ้นเพื่ออธิบายบทบาท หน้าที่การทำงาน และความสำคัญของทุกแพ็กเกจที่ระบุอยู่ในไฟล์ `package.json` ของทั้ง 3 Widgets ใน Repository นี้ ได้แก่ **pwbComboBox**, **pwbCustomizeContainerDataView** และ **pwbDatePicker** ครอบคลุมทั้งส่วนของ Dependencies, devDependencies รวมถึงระบบความปลอดภัยและการควบคุมเวอร์ชันด้วย Overrides & Resolutions

---

## 📌 บทนำ: ความแตกต่างของสเตจในโปรเจค Pluggable Widget

ในการพัฒนา Mendix Pluggable Widgets การแบ่งสัดส่วนแพ็กเกจในไฟล์ `package.json` มีผลอย่างยิ่งต่อขนาดและความเร็วของการทำงานของ Widget บนเครื่องของผู้ใช้จริง:

1. **dependencies**: แพ็กเกจที่จำเป็นต้องโหลดและรันบนเว็บบราวเซอร์ของผู้ใช้งานจริง (Client-side Production Runtime) ทุกครั้งที่มีการใช้งาน Widget
2. **devDependencies**: แพ็กเกจที่ใช้ **เฉพาะขั้นตอนการพัฒนา การบิลด์ และการทดสอบระบบ** ในเครื่องของนักพัฒนาเท่านั้น จะไม่ถูกส่งเข้าไปใน Mendix Runtime (ยกเว้นโค้ดที่ถูกคอมไพล์และ Bundled รวมผ่าน Webpack แล้ว)
3. **overrides & resolutions**: การบังคับควบคุมเวอร์ชันของแพ็กเกจย่อยที่อยู่ภายใต้แพ็กเกจหลักอีกที (Nested Dependencies) เพื่อป้องกันปัญหาความขัดแย้งของเวอร์ชัน (Version Mismatch) และรับประกันว่าระบบทั้งหมดทำงานภายใต้รันไทม์ React เวอร์ชันเดียวกัน

## 🔍 เจาะลึกการทำงานและความสำคัญของแต่ละประเภท

### 1. Dependencies (ใช้รันในสภาพแวดล้อมจริง)

#### 🔹 `classnames` (v2.2.6)
* **การติดตั้ง**: มีใน `pwbComboBox` และ `pwbDatePicker`
* **การทำงาน (How it works)**:
  * เป็น JavaScript utility ขนาดเล็กที่ใช้สำหรับช่วยต่อสายอักขระ (String Concatenation) ของคลาส CSS แบบมีเงื่อนไขที่เข้าใจง่าย 
  * แทนการต่อสตริงด้วยเงื่อนไขยุ่งยาก เช่น `className={"btn " + (isActive ? "btn-active" : "")}` จะเปลี่ยนมาใช้ `className={classNames('btn', { 'btn-active': isActive })}` แทน
* **ความสำคัญต่อโปรเจค**:
  * **จัดการ UI States ได้สะอาดและเป็นระเบียบ**: ทั้ง ComboBox และ DatePicker มีสถานะของ UI เยอะมาก (เช่น hover, active, focus, disabled, error, invalid) ไลบรารีนี้ช่วยให้เราสลับคลาสของ CSS ได้อย่างคล่องตัวและลดความผิดพลาดจากเว้นวรรค (space) ระหว่างคลาส CSS
  * **เพิ่มความเสถียรและบำรุงรักษาง่าย**: ลดความซับซ้อนของโค้ดสไตล์เลย์เอาต์ ปลอดภัยเมื่อต้องรับพารามิเตอร์แบบ nullable

### 2. devDependencies (ใช้พัฒนาและบิลด์)

#### 🔹 `@mendix/pluggable-widgets-tools` (v11.6.0)
* **การทำงาน (How it works)**:
  * เป็น Toolchain ของ Mendix ที่ห่อหุ้ม Webpack, Gulp และเครื่องมือบิลด์อื่นๆ ไว้ภายใน เพื่อสร้างและบีบอัดโค้ดให้อยู่ในฟอร์แมต `.mpk` (Mendix Package)
* **ความสำคัญต่อโปรเจค**:
  * **หัวใจของการจัดเตรียมชิ้นงาน**: ถ้าไม่มีตัวนี้ จะไม่สามารถแปลงไฟล์ TypeScript/JSX ไปเป็นโค้ดที่สามารถนำไปอัปโหลดเข้าสู่หน้า Mendix Studio Pro หรือรันบนเครื่อง Mendix Client API ได้

#### 🔹 `vite` (v5.0.0) และ `@vitejs/plugin-react` (v4.2.0)
* **การทำงาน (How it works)**:
  * ระบบ Development Server ยุคใหม่ที่ทำงานได้เร็วมากโดยใช้ Native ESM และช่วยคอมไพล์โค้ด React ร่วมกับฟีเจอร์ Hot Module Replacement (HMR) 
* **ความสำคัญต่อโปรเจค**:
  * **สภาพแวดล้อมจำลองแบบรวดเร็ว (Playground)**: ทำให้นักพัฒนาสามารถทดลองเล่น แก้ไขข้อผิดพลาด และทดสอบพฤติกรรมของ Component แบบอิสระได้อย่างรวดเร็วในเสี้ยววินาที โดยไม่ต้องรอการรัน Mendix App ทั้งระบบซึ่งมีความล่าช้า

#### 🔹 `@types/big.js` (v6.0.2) และ `big.js` (v6.2.1)
* **การทำงาน (How it works)**:
  * `big.js` ช่วยจัดการคำนวณคณิตศาสตร์ทศนิยมความละเอียดสูง (Decimal) ของ JavaScript ส่วน `@types/big.js` จัดเตรียมข้อมูลโครงสร้างการประกาศประเภทตัวแปรให้ TypeScript
* **ความสำคัญต่อโปรเจค**:
  * **ความแม่นยำสูง**: ในระบบ Mendix ข้อมูลประเภท Decimal จะถูกดึงและส่งออกผ่านอ็อบเจกต์ Big เสมอ การติดตั้งสองแพ็กเกจนี้ช่วยจำลองและพัฒนาตรรกะคำนวณให้แม่นยำ ป้องกันบั๊กเกี่ยวกับตัวเลขทศนิยมที่ผิดพลาด และมี Autocomplete แนะนำโค้ดระหว่างเขียน

#### 🔹 `vitest` (v4.1.7)
* **การทำงาน (How it works)**:
  * เฟรมเวิร์กสำหรับรันระบบทดสอบอัตโนมัติ (Unit Testing) ที่ทำงานผสานร่วมกับ Vite
* **ความสำคัญต่อโปรเจค**:
  * **การันตีเสถียรภาพ**: รันเทสไฟล์ใน `pwbDatePicker` เพื่อตรวจสอบการแปลงฟอร์แมตวันที่ตามท้องถิ่น ป้องกันการอัปเกรดโค้ดแล้วทำให้ลอจิกพังเสียหาย (Regression Testing)

### 3. Overrides & Resolutions (การบังคับควบคุมเวอร์ชัน)

ใน `package.json` ของทั้ง 3 Widgets มีการประกาศบล็อก `"overrides"` และ `"resolutions"` เพื่อบังคับล็อกเวอร์ชันของ React:
```json
"overrides": {
  "react": "^19.0.0",
  "react-dom": "^19.0.0",
  "@types/react": "^19.0.0",
  "@types/react-dom": "^19.0.0"
}
```

* **ทำไมต้องทำแบบนี้? (Why it is important?)**:
  1. **บังคับใช้ React 19 ยุคใหม่**: เครื่องมือบิลด์อิงค์มาตรฐานของ Mendix บางตัวอาจบังคับติดตั้ง React 17 หรือ 18 เป็นค่าดีฟอลต์ (Default Dependency) การใส่สิ่งนี้จะเป็นการสั่งการให้ตัวจัดการแพ็กเกจ (NPM หรือ Yarn) เพิกถอนเวอร์ชันเก่าของ sub-dependencies ทั้งหมด และบังคับดาวน์โหลดและรันด้วย React เวอร์ชัน 19 แทน
  2. **ความพร้อมเพรียงข้ามระบบ**: รับประกันว่าไลบรารีย่อยทั้งหมดใน Widget จะใช้ React 19 เพื่อให้เข้ากันได้กับสภาพแวดล้อม Client Runtime ของ Mendix 10+
  3. **หลีกเลี่ยงข้อผิดพลาดประเภท Multiple React Instance**: หากในเวิร์กสเปซมี React มากกว่าหนึ่งเวอร์ชันซ้อนกัน React จะแสดง Error และหยุดทำงานทันที (React runtime conflicts) การทำ Overrides/Resolutions จึงเป็นการแก้ปัญหานี้ตั้งแต่ต้นลม

## 💡 สรุปประโยชน์ของสถาปัตยกรรมนี้ต่อโปรเจค
การจัดระเบียบโครงสร้างแพ็กเกจร่วมกันอย่างมีระบบข้ามทั้ง 3 Widgets นี้ ส่งผลดีต่อโปรเจคดังนี้:
1. **ประสิทธิภาพสูงสุด (Optimal Performance)**: ตัวรันไทม์จริงบนเบราว์เซอร์ลูกค้าจะไม่มีการโหลดเครื่องมือพัฒนาขนาดยักษ์ ทำให้ตัว Widget ประมวลผลได้อย่างรวดเร็วและใช้แบนด์วิดท์น้อยที่สุด
2. **การทำงานร่วมกันที่เป็นมาตรฐานเดียวกัน (Uniform Development Stack)**: นักพัฒนาทุกคนสามารถใช้สคริปต์คำสั่ง และเฟรมเวิร์กการเขียนโค้ดและทดสอบในมาตรฐานเดียวกัน ช่วยลด Cognitive Load ในการดูแลและพัฒนา Widget หลายตัวในคราวเดียวกัน
3. **ควบคุมความเข้ากันได้ของแพลตฟอร์มอย่างมั่นคง**: การล็อกเวอร์ชันผ่าน Overrides & Resolutions ช่วยตัดปัญหาโค้ดของ Library ขัดแย้งกับเวอร์ชัน Mendix API ที่คอยอัปเดตอย่างต่อเนื่อง
