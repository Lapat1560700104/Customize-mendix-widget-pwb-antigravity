# คู่มืออธิบายการทำงานและความสำคัญของ devDependencies ในโปรเจค
*(devDependencies Technical Reference Guide)*

เอกสารฉบับนี้อธิบายบทบาท หน้าที่การทำงาน และความสำคัญของแพ็กเกจที่ระบุอยู่ในส่วนของ `devDependencies` (Development Dependencies) ของทั้ง 3 Widgets ใน Repository นี้ ได้แก่ **pwbComboBox**, **pwbCustomizeContainerDataView** และ **pwbDatePicker** เพื่อให้นักพัฒนาเข้าใจการทำงานของเครื่องมือบิลด์ (Build Tooling) และระบบการทดสอบได้อย่างถูกต้อง

---

## 📌 บทนำ: devDependencies ใน Mendix Pluggable Widget คืออะไร?

ในการพัฒนา Mendix Pluggable Widgets แพ็กเกจจะถูกแบ่งออกเป็น 2 ประเภทหลักในไฟล์ `package.json`:
1. **dependencies**: แพ็กเกจที่จำเป็นต้องทำงานใน Runtime บนเว็บเบราว์เซอร์ของผู้ใช้งานจริง (Client-side Runtime) เมื่อนำ Widget ไปใช้งานบน Mendix App เช่น ไลบรารีจัดการ CSS `classnames`
2. **devDependencies**: แพ็กเกจที่ใช้ **เฉพาะขั้นตอนการพัฒนาและการทดสอบในเครื่องของนักพัฒนาเท่านั้น** เช่น ตัวช่วยแปลงรหัส (Transpilers), เครื่องมือจัดรูปแบบโค้ด (Linters), และระบบทดสอบ (Test Suites) แพ็กเกจเหล่านี้**จะไม่ถูกรวมเข้าไปในไฟล์ `.mpk` ที่ใช้จริงในระบบ Mendix App** (ยกเว้นบางกรณีพิเศษที่โค้ดเหล่านั้นถูก Bundled รวมผ่าน Webpack ของตัว Build Tool) ส่งผลให้ตัว Widget มีขนาดเล็กและมีประสิทธิภาพสูง

## 🔍 เจาะลึกการทำงานและความสำคัญของแต่ละแพ็กเกจ

### 1. `@mendix/pluggable-widgets-tools` (v11.6.0)
* **การติดตั้ง**: ติดตั้งในทุก Widgets (หัวใจหลักของโปรเจค)
* **การทำงาน (How it works)**:
  * เป็นชุดเครื่องมือสั่งการอย่างเป็นทางการที่พัฒนาโดยทีมงาน Mendix ซึ่งได้รวมเครื่องมือเบื้องหลังยอดนิยมอย่าง Webpack, ESLint, Prettier, TypeScript Compiler และ Gulp เข้ามารวมไว้เป็นคำสั่งเดียว
  * ทำหน้าที่ควบคุมคำสั่ง (Scripts) สำคัญในไฟล์ `package.json` ของแต่ละ Widget ได้แก่:
    * `start` และ `dev` (`pluggable-widgets-tools start:web`): รัน Dev Server และเฝ้าดูการเปลี่ยนแปลงของไฟล์เพื่ออัปเดตแบบ Dynamic
    * `build` (`pluggable-widgets-tools build:web`): แปลงไฟล์ TypeScript/JSX ไปเป็น JavaScript (ES5/ES6) ที่เบราว์เซอร์ทั่วไปสามารถประมวลผลได้
    * `lint` และ `lint:fix`: ตรวจสอบและแก้ไขสไตล์การเขียนโค้ดตามมาตรฐานของ Mendix
    * `release` (`pluggable-widgets-tools release:web`): รันกระบวนการย่อโค้ด (Minify/Uglify), รวมไฟล์ Asset, และทำการบีบอัดแพ็คเกจให้อยู่ในฟอร์แมต `.mpk` (Mendix Package)
* **ความสำคัญต่อโปรเจค**:
  * **จำเป็นอย่างยิ่งสำหรับการแปลงไฟล์**: หากไม่มีเครื่องมือนี้ นักพัฒนาจะไม่สามารถสร้างไฟล์ `.mpk` เพื่อนำเข้าไปติดตั้งและใช้งานใน Mendix Studio Pro ได้
  * **การันตีความเข้ากันได้**: ควบคุมให้โครงสร้างไฟล์ คุณสมบัติ XML และโค้ด JS ที่บิลด์ออกมาสามารถทำงานร่วมกับ Mendix Client API เวอร์ชันต่างๆ ได้อย่างไร้รอยต่อ

> [!NOTE]
> เวอร์ชัน `^11.6.0` เป็นเวอร์ชันที่ออกแบบมารองรับการพัฒนาบนสภาพแวดล้อม Mendix 10 เป็นต้นไป ช่วยให้การจัดการ Bundle ของ React 18/19 และโค้ดสมัยใหม่มีความเสถียรยิ่งขึ้น

### 2. `vite` (v5.0.0)
* **การติดตั้ง**: ติดตั้งในทุก Widgets
* **การทำงาน (How it works)**:
  * เป็น build tool ยุคใหม่ที่ทำงานได้เร็วมาก โดยอาศัย Native ES Modules (ESM) บนเบราว์เซอร์ในการโหลดโค้ดระหว่างการพัฒนา แทนการทำ Bundling ทั้งหมดตั้งแต่แรกแบบ Webpack
  * ถูกระบุให้ทำงานร่วมกับคำสั่ง `"playground": "vite playground"` ในโปรเจค
* **ความสำคัญต่อโปรเจค**:
  * **ระบบจำลองพัฒนาที่รวดเร็ว (Fast Feedback Loop)**: การพัฒนา Widget บน Mendix แบบดั้งเดิมจำเป็นต้องเปิดโปรแกรม Mendix Studio Pro และรันโปรเจค Mendix ทั้งหมด ซึ่งใช้ทรัพยากรเครื่องและเวลาในการเปิดค่อนข้างสูง การใช้ Vite ช่วยเปิด "Playground" ขึ้นมาทดสอบตัว UI/UX ของ Widget ได้แบบ Standalone รวดเร็วทันใจ
  * **Hot Module Replacement (HMR)**: เมื่อทำการแก้ไขโค้ด Vite จะทำการอัปเดตเฉพาะ Component นั้นๆ บนหน้าจอเบราว์เซอร์ทันทีในเสี้ยววินาทีโดยไม่ต้องโหลดหน้าเว็บใหม่ทั้งหมด ช่วยประหยัดเวลาของนักพัฒนาลงได้อย่างมหาศาล

---

### 3. `@vitejs/plugin-react` (v4.2.0)
* **การติดตั้ง**: ติดตั้งในทุก Widgets
* **การทำงาน (How it works)**:
  * ปลั๊กอินหลักที่เชื่อมต่อระหว่าง Vite และ React เพื่อช่วยปรับแต่งขั้นตอนการคอมไพล์ JSX และเปิดใช้ฟีเจอร์ React Fast Refresh
* **ความสำคัญต่อโปรเจค**:
  * **รองรับ React บน Vite**: ทำหน้าที่แปลงไวยากรณ์ TSX/JSX ใน Playground ให้เป็นโค้ด React ที่เบราว์เซอร์เข้าใจ
  * **รักษาสถานะ Component**: ช่วยให้สามารถรีโหลดส่วนของหน้าจอที่แก้ไขได้โดยไม่ล้างสถานะ (State) เก่าของ Component ทำให้นักพัฒนาสามารถทดสอบกรณีการกรอกข้อมูลหรือการคลิกเลือกค่าต่างๆ ได้อย่างต่อเนื่อง

---

### 4. `@types/big.js` (v6.0.2)
* **การติดตั้ง**: ติดตั้งในทุก Widgets
* **การทำงาน (How it works)**:
  * เป็นไฟล์คำอธิบายประเภทตัวแปร (TypeScript Declaration Files) สำหรับไลบรารีคณิตศาสตร์ทศนิยมที่มีความละเอียดสูง (`big.js`)
* **ความสำคัญต่อโปรเจค**:
  * **Type Safety & Intellisense**: ช่วยให้ระบบวิเคราะห์โค้ดในตัวแก้ไข (เช่น VS Code) เข้าใจโครงสร้างเมธอดของอ็อบเจกต์ Big เช่น `.plus()`, `.minus()`, `.times()` และแสดงระบบแนะนำโค้ดอัตโนมัติ (Autocomplete) ป้องกันความผิดพลาดจากการพิมพ์ชื่อฟังก์ชันหรือการส่งอาร์กิวเมนต์ผิดรูปแบบ

### 5. `big.js` (v6.2.1)
* **การติดตั้ง**: ติดตั้งใน **pwbCustomizeContainerDataView** (พบภายใต้ devDependencies)
* **การทำงาน (How it works)**:
  * ไลบรารีคำนวณคณิตศาสตร์ทศนิยมความแม่นยำสูง เพื่อแก้ปัญหาความไม่ถูกต้องทางทศนิยมของ JavaScript ทั่วไป (เช่น `0.1 + 0.2 === 0.30000000000000004`)
* **ความสำคัญต่อโปรเจค**:
  * **การจำลองค่า Decimal ใน Mendix**: ในตัวแปรประเภททศนิยม (Decimal) ของ Mendix Pluggable Widget จะส่งข้อมูลเข้ามาในรูปของอ็อบเจกต์ `Big` เสมอ การมีไลบรารีนี้ใน `devDependencies` ของตัว Widget ช่วยสนับสนุนขั้นตอนการทดสอบ (Testing) หรือจำลอง Mock Data ในส่วนของ Playground ให้มีพฤติกรรมเหมือนกับตอนทำงานอยู่บน Mendix จริงๆ

---

### 6. `vitest` (v4.1.7)
* **การติดตั้ง**: ติดตั้งเฉพาะใน **pwbDatePicker**
* **การทำงาน (How it works)**:
  * เฟรมเวิร์กสำหรับรันการทดสอบระดับย่อย (Unit Testing) ที่ทำงานร่วมกับจุดเด่นด้านความเร็วของ Vite ได้อย่างดีเยี่ยม
  * ทำหน้าที่สแกนไฟล์ทดสอบ (เช่น `*.spec.ts` หรือ `*.test.tsx`) และสั่งรันผลลัพธ์ผ่านคำสั่ง `"test": "vitest run"`
* **ความสำคัญต่อโปรเจค**:
  * **การรันเทสอย่างรวดเร็ว**: รองรับการตรวจสอบการทำงานของฟังก์ชันเลือกวันที่ (Date-picking logic), การจัดฟอร์แมตวันที่ตามแต่ละภาษา (Localization Formats), และการตรวจสอบเงื่อนไขขอบเขตเวลา (Min/Max Date Validations) ซึ่งเป็นลอจิกที่มีความสลับซับซ้อน
  * **รับประกันคุณภาพของโค้ด**: ป้องกันไม่ให้การอัปเกรดโค้ดหรือการปรับแต่งลอจิกย่อยไปกระทบโครงสร้างการทำงานเดิมที่ถูกต้องอยู่แล้ว (Regression Prevention)

## ⚙️ ขั้นตอนการรันและบทบาทของเครื่องมือเหล่านี้ในการพัฒนา

เมื่อคุณทำงานภายในโปรเจค Monorepo นี้ เครื่องมือ `devDependencies` จะเข้ามารับช่วงต่อตามคำสั่งดังนี้:

```mermaid
flowchart TD
    A[เริ่มพัฒนา Widget] --> B{ต้องการทำอะไร?}
    
    B -->|ทดสอบ Component แยกต่างหาก| C[npm run playground]
    C -->|เบื้องหลังทำงาน| C1[vite & @vitejs/plugin-react]
    
    B -->|เขียนเทส / รันระบบทดสอบ| D[npm run test]
    D -->|เบื้องหลังทำงาน| D1[vitest]
    
    B -->|ตรวจสอบสไตล์และรูปแบบโค้ด| E[npm run lint]
    E -->|เบื้องหลังทำงาน| E1[@mendix/pluggable-widgets-tools]
    
    B -->|บิลด์โค้ดเพื่อใช้งานใน Mendix| F[npm run release / build]
    F -->|เบื้องหลังทำงาน| F1[@mendix/pluggable-widgets-tools]
```

1. **ขั้นตอนการโค้ดดิ้ง**: โค้ดจะได้รับการตรวจสอบความปลอดภัยและอำนวยความสะดวกด้วย `@types/big.js`
2. **ขั้นตอนจำลองบนบราวเซอร์**: เรียกใช้งาน `vite` + `@vitejs/plugin-react` และ `big.js` (สำหรับจำลองข้อมูล) ผ่านคำสั่ง `npm run playground`
3. **ขั้นตอนการทดสอบ**: รัน `vitest` เพื่อตรวจสอบความถูกต้องของลอจิกตัวแปรผ่าน `npm run test`
4. **ขั้นตอนการแพ็คเกจเพื่อส่งต่อ**: เรียกใช้งาน `@mendix/pluggable-widgets-tools` บิลด์และแพ็คไฟล์ออกมาระดับโปรดักชัน เพื่อจัดเตรียมส่งมอบเป็นไฟล์ `.mpk` ในลำดับถัดไป

## 💡 สรุปประโยชน์ของสถาปัตยกรรมนี้ต่อโปรเจค
การตั้งค่า `devDependencies` ที่ชัดเจนและมีความสอดคล้องกันข้ามทั้ง 3 Widgets นี้ ช่วยส่งผลดีต่อโปรเจคโดยรวมดังนี้:
1. **ลดภาระในการโหลด (Zero-Overhead in Production)**: มั่นใจได้ว่าไลบรารีขนาดใหญ่อย่าง `vite` หรือ `vitest` จะไม่ถูกส่งเข้าไปยัง Mendix runtime ส่งผลให้ตัวแอปพลิเคชันหลักของลูกค้าโหลดหน้าเว็บได้รวดเร็วที่สุด
2. **เครื่องมือพัฒนาที่เป็นเอกภาพ**: ไม่ว่าจะเลือกพัฒนาชิ้นงานไหนในทั้ง 3 Widgets นักพัฒนาสามารถใช้คำสั่งชุดเดียวกัน สไตล์การเขียนโค้ดมาตรฐานเดียวกัน และรูปแบบของเทสเฟรมเวิร์กเดียวกันได้ทั้งหมด ช่วยลดขั้นตอนการศึกษาโค้ดและการเริ่มงานใหม่
