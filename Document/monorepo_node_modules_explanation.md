# คู่มืออธิบายโฟลเดอร์ node_modules ในระดับ Root (Monorepo Node Modules Guide)

เอกสารฉบับนี้จัดทำขึ้นเพื่ออธิบายเหตุผลความจำเป็น โครงสร้าง และกลไกการทำงานของโฟลเดอร์ `node_modules` ที่ปรากฏอยู่ในโฟลเดอร์หลัก (Root) ของโปรเจกต์พัฒนา Mendix Pluggable Widgets นี้

---

## 📌 บทนำ: ทำไมถึงมี node_modules ที่ Root หลัก?

ในโปรเจกต์นี้ มีการจัดโครงสร้างโค้ดแบบ **Monorepo (Multi-Widget)** เพื่อจัดการและพัฒนาหลาย Widgets ร่วมกันใน Repository เดียว โดยใช้ฟีเจอร์ **npm Workspaces**

เมื่อทำการรันคำสั่งติดตั้ง Dependencies (เช่น `npm install`) ระบบของ npm จะสร้างโฟลเดอร์ `node_modules` ขึ้นที่ระดับ Root หลักด้วยแนวคิดที่เรียกว่า **"Dependency Hoisting" (การดึงชุดไลบรารีขึ้นมาไว้ด้านบน)**

> [!NOTE]
> การมี `node_modules` ที่ Root เป็นพฤติกรรมมาตรฐานของระบบ npm Workspaces และ Yarn Workspaces ซึ่งได้รับการออกแบบมาเพื่อช่วยบริหารจัดการโปรเจกต์ขนาดใหญ่ที่มีโปรเจกต์ย่อยซ้อนอยู่ภายใน

---

## ⚙️ กลไกการทำงานของ npm Workspaces (Hoisting)

เมื่อคุณเปิดดูโครงสร้างในไฟล์ [package.json](file:///Users/lapat.ta/Desktop/ETC%20Project/Customize-mendix-widget-pwb-antigravity/package.json) ของโฟลเดอร์ Root จะพบคำสั่งดังนี้:

```json
{
  "name": "pwb-widgets-monorepo",
  "private": true,
  "workspaces": [
    "pwb*"
  ]
}
```

ด้วยการตั้งค่านี้ เมื่อมีการรันกระบวนการติดตั้งไลบรารี:

1. **การรวมศูนย์การดาวน์โหลด**: npm จะเข้าไปอ่านไฟล์ `package.json` ของทุกโฟลเดอร์ย่อยที่ขึ้นต้นด้วย `pwb*` เช่น [pwbComboBox/package.json](file:///Users/lapat.ta/Desktop/ETC%20Project/Customize-mendix-widget-pwb-antigravity/pwbComboBox/package.json) และ [pwbDatePicker/package.json](file:///Users/lapat.ta/Desktop/ETC%20Project/Customize-mendix-widget-pwb-antigravity/pwbDatePicker/package.json)
2. **การทำ Hoisting**: แทนที่ npm จะติดตั้งไลบรารีซ้ำๆ แยกกันในแต่ละโฟลเดอร์ย่อย npm จะนำไลบรารีเหล่านั้นมาเก็บรวมกันไว้ที่โฟลเดอร์ `node_modules` ของ Root หลักเพียงที่เดียว
3. **การสร้าง Symlinks (ลิงก์เสมือน)**: สำหรับตัวโปรเจกต์ย่อยเอง npm จะสร้างโฟลเดอร์ `node_modules` ย่อยขึ้นมาในโฟลเดอร์นั้นๆ แต่ภายในจะเก็บเฉพาะ **Symlinks** (ลิงก์ทางลัดอ้างอิง) กลับมาชี้ยังโฟลเดอร์ `node_modules` ตัวจริงที่อยู่ด้านนอกสุด

## 📂 ทำไมในโฟลเดอร์ของ Widget แต่ละตัว ยังคงมีโฟลเดอร์ node_modules อยู่?

หลายคนอาจสงสัยว่า ในเมื่อเราทำการดึง (Hoist) เอา Dependencies ทั้งหมดไปไว้ที่ Root แล้ว ทำไมเมื่อเข้าไปดูในโฟลเดอร์ของ Widget ย่อย (เช่น `pwbComboBox/node_modules`) จึงยังมีโฟลเดอร์นี้อยู่เช่นกัน?

สาเหตุและหน้าที่สำคัญของโฟลเดอร์ `node_modules` ภายในโฟลเดอร์ย่อยแต่ละ Widget มีดังนี้:

### 1. เป็นช่องทางเข้าถึงผ่าน Symlinks (ลิงก์เสมือน)

ในระดับโครงสร้างไฟล์ (Filesystem) โฟลเดอร์ `node_modules` ย่อยไม่ได้เก็บชุดโค้ดของไลบรารีจริงๆ แต่จะมี **Symlink (หรือทางลัด)** ที่ระบบสร้างขึ้นมาอัตโนมัติ เพื่อชี้กลับไปยังโฟลเดอร์ `node_modules` หลักที่ระดับ Root

* วิธีนี้ช่วยให้โปรแกรมแก้ไขโค้ด (เช่น VS Code / Cursor) และเครื่องมือคอมไพล์บิลด์ (Webpack/Vite/TypeScript) ยังคงมองเห็น ค้นหา และเรียกใช้ Dependencies ต่างๆ (เช่น `React` หรือ `classnames`) ผ่าน Path ท้องถิ่นของตัวเองตามมาตรฐานปกติของ Node.js ได้อย่างราบรื่น

### 2. การลงทะเบียนเชื่อมโยงระหว่าง Widget ด้วยกันเอง

เมื่อใช้ระบบ npm Workspaces ทุก Widget ย่อยในโปรเจกต์นี้จะได้รับการลงทะเบียนและสร้าง Symlinks เชื่อมต่อกันเองไว้ในโฟลเดอร์ `node_modules` ของกันและกัน

* ตัวอย่างเช่น ในโฟลเดอร์ `pwbComboBox/node_modules/` จะมีลิงก์เสมือนที่ชี้ไปยังโฟลเดอร์ของ Widget อื่นๆ เช่น `pwbDatePicker`
* สิ่งนี้ทำให้นักพัฒนาสามารถเขียนโค้ดเรียกใช้งานหรือนำเข้า (import) โค้ดข้ามกันระหว่าง Widgets ได้ง่าย สะดวก และปลอดภัย ราวกับเรียกใช้ Library ภายนอกทั่วไป โดยไม่จำเป็นต้องสั่ง Publish ขึ้นสู่ระบบ npm registry จริงๆ

### 3. การรองรับ Library เฉพาะตัวที่ไม่ถูกดึงขึ้นไป (Unhoisted Dependencies)

ในบางกรณี หาก Widget ตัวใดตัวหนึ่งใช้ Library เวอร์ชันที่ไม่เหมือนกับตัวอื่น หรือมีการระบุเงื่อนไขไม่ให้ยกแพ็กเกจขึ้นไปด้านบน (Unhoist) npm จะดาวน์โหลดแพ็กเกจจำเพาะนั้นมาบันทึกไว้ใน `node_modules` ของ Widget ตัวนั้นๆ โดยตรง เพื่อป้องกันไม่ให้เวอร์ชันตีกันกับ Widget ตัวอื่นที่ระดับ Root

## 📊 เปรียบเทียบการเก็บ node_modules แบบเดี่ยว vs แบบรวมศูนย์ (Root)

| หัวข้อเปรียบเทียบ | แบบรวมศูนย์ไว้ที่ Root (Monorepo - ปัจจุบัน) | แบบแยกติดตั้งเองรายโฟลเดอร์ย่อย |
| :--- | :--- | :--- |
| **พื้นที่ฮาร์ดดิสก์** | **ประหยัดอย่างมาก**: ไลบรารีที่ซ้ำกัน (เช่น `@mendix/pluggable-widgets-tools`, `React`, `TypeScript`) จะถูกดาวน์โหลดและบันทึกเพียงครั้งเดียว | **เปลืองพื้นที่สูง**: แต่ละ Widget จะต้องดาวน์โหลดชุดไลบรารีขนาดใหญ่แยกกันทั้งหมดในทุกโฟลเดอร์ |
| **เวลาในการติดตั้ง** | **เร็วขึ้น**: ติดตั้งคำสั่งเดียวได้ครบทุก Widget และใช้ Cache ร่วมกันได้ | **ช้า**: ต้องเปลี่ยนโฟลเดอร์ (`cd`) เข้าไปสั่ง `npm install` ทีละโฟลเดอร์ |
| **ความสอดคล้องของเวอร์ชัน** | **ดีเยี่ยม**: มั่นใจได้ว่าทุก Widget ใช้ไลบรารีและเครื่องมือพัฒนา (Build Tools) เวอร์ชันเดียวกัน | **ยากต่อการควบคุม**: เครื่องมือของ Widget แต่ละตัวอาจมีเวอร์ชันต่างกันจนส่งผลให้บิลด์ไม่ผ่าน |
| **การจัดการ Lockfile** | **มีไฟล์เดียว**: มีไฟล์ `package-lock.json` แค่ไฟล์เดียวที่ Root สำหรับล็อกโครงสร้าง Dependency ทั้งระบบ | **มีหลายไฟล์**: แต่ละ Widget จะมี lockfile แยกกัน ทำให้ยากต่อการรีวิวความเสถียร |

## 💡 ประโยชน์สำคัญต่อการพัฒนา Mendix Pluggable Widgets

1. **การควบคุมการบิลด์ข้าม Workspaces**: ช่วยให้เราสามารถสร้างคำสั่งลัด (Shortcuts) ใน Root เพื่อสั่งการพร้อมกันได้ เช่น `npm run build:all` ซึ่งคำสั่งจะไปเรียกใช้เครื่องมือบิลด์ที่ถูกติดตั้งไว้ส่วนกลางที่ Root นี้
2. **การทำงานร่วมกันภายใต้ React 19**: เนื่องจาก Mendix 10+ ขึ้นไปเน้นใช้ React 19 การรวมโค้ดและบังคับใช้ Overrides (Resolutions) ไว้ที่จุดศูนย์กลางระดับ Root จะทำให้ไม่มีไลบรารี React เวอร์ชันอื่นปะปนเข้ามาจนทำให้ Widget พังเสียหาย

> [!IMPORTANT]
> **ข้อพึงระวัง:** ห้ามลบโฟลเดอร์ `node_modules` หรือไฟล์ `package-lock.json` ที่ Root หลักออก เพราะจะทำให้ Symlinks ใน Widget ย่อยเสียหายและไม่สามารถบิลด์ไฟล์ `.mpk` ได้ หากต้องการล้างระบบเพื่อติดตั้งใหม่ ให้ลบโฟลเดอร์ `node_modules` ทั้งที่ Root และโฟลเดอร์ย่อยออกพร้อมกันแล้วรัน `npm install` ที่ Root หลักใหม่อีกครั้ง
