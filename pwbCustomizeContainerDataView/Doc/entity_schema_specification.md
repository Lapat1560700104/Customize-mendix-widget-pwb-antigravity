# 🗄️ คู่มือข้อกำหนดโครงสร้าง Entity และการออกแบบฐานข้อมูล (Entity Schema Specification)

คู่มือนี้มีวัตถุประสงค์เพื่อระเบียบวิธีและกำหนดแนวทางการออกแบบ **Mendix Entity Schema** ให้ถูกต้อง สอดคล้องกับการใช้งาน Widget **PwbCustomizeContainerDataView** เพื่อประสิทธิภาพในการจัดเรียงและบันทึกลำดับแบบลากวาง

---

## 📌 1. โครงสร้างความสัมพันธ์ระดับ Entity (High-Level Entity Relationship)

ในการใช้ Widget นี้ ดาต้าโมเดลของคุณจะต้องประกอบด้วย **สองส่วนที่มีความสัมพันธ์กัน** ดังแผนภาพด้านล่างนี้:

```
┌──────────────────────────────────────┐          ┌──────────────────────────────────────┐
│        Parent/Context Entity         │          │         Child/Item Entity            │
│  (เช่น TaskBoard, KanbanColumn)       │          │  (เช่น Task, OptionItem, Lesson)     │
├──────────────────────────────────────┤          ├──────────────────────────────────────┤
│ 🔑 sortedAttribute (String/Unlimited)│◄─────────┼─ Parent_Child (Association)          │
└──────────────────────────────────────┘          └──────────────────────────────────────┘
```

* **Parent Entity (Context)**: ทำหน้าที่เป็น "ที่อยู่อาศัยหลัก" หรือกรอบครอบ (Container context) ซึ่งมี Attribute สำหรับเก็บบันทึกลำดับของข้อมูล
* **Child Entity (DataSource)**: เป็น "ไอเทมย่อย" แต่ละตัวที่จะปรากฏขึ้นมาเป็นการ์ดในลิสต์รายการ และถูกจับลากสลับตำแหน่งได้

---

## 🔑 2. ข้อกำหนดของ Attributes (Attribute Requirements)

### A. สำหรับ Parent Entity (Context Entity)

เป็น Entity ที่ครอบครอบหน้าจอหลัก หรือ DataView ที่ Widget ตัวนี้วางอยู่

| Attribute | ประเภทข้อมูล (Type) | สถานะ | คำอธิบายความจำเป็น |
| :--- | :--- | :---: | :--- |
| **Sorted IDs (เช่น `SortedTaskIds`)** | **String (Unlimited)** | 🚨 **บังคับ (Mandatory)** | **หัวใจสำคัญของระบบ**: ใช้เก็บสายอักขระความยาวสูงที่เป็น GUIDs ของไอเทมย่อยมาต่อกันด้วยเครื่องหมายจุลภาค (เช่น `123456789012,123456789013,123456789014`) หากไม่มีฟิลด์นี้ Widget จะไม่สามารถเรียงลำดับหรือบันทึกตำแหน่งล่าสุดได้ |

### B. สำหรับ Child Entity (Item Entity)

เป็น Entity ของลิสต์รายการที่ดึงมาจาก Datasource เพื่อส่งมาแสดงผลและสลับลำดับ

* **ไม่บังคับ Attribute ใดๆ เป็นพิเศษระดับบิวด์อิน**: เนื่องจาก Widget ตัวนี้ทำงานในรูปแบบ **Dropdown / Nested Widget** ที่ดึงเอา GUID ภายในระบบของแต่ละ Object (Mendix Internal ID / GUID) มาใช้งานโดยตรง
* **Attributes ทั่วไปที่แนะนำ**: ฟิลด์เก็บข้อมูลการแสดงผล เช่น `Title` (String), `Description` (String), `CreatedDate` (DateTime), `Status` (Enum) เป็นต้น สำหรับผูกแสดงผลบนการ์ดแถวรายการ

## 🚨 3. สิ่งสำคัญที่ "บังคับ" ต้องตั้งค่าให้ถูกต้อง

1. **สิทธิ์การเขียนข้อมูลของ Sorted Attribute**: 
   สิทธิ์การเข้าถึง (Access Rules) ของ User Role ที่ใช้งาน จะต้องได้รับอนุญาตให้ทำการเขียนข้อมูล (**Read/Write**) ลงบน Attribute ตัวที่เป็น Sorted IDs เสมอ หากเป็น Read-only ตัว Widget จะลากสลับตำแหน่งได้บนหน้าเว็บชั่วคราว แต่จะแจ้งเตือนและไม่เซฟข้อมูลตำแหน่งลงสู่ระบบฐานข้อมูล
2. **ผูก DataSource ความสัมพันธ์ให้สอดคล้องกัน**:
   ไอเทมย่อยของ Child Entity ที่ดึงมาแสดงผลใน `itemsSource` จะต้องดึงข้อมูลผ่าน Association ที่เชื่อมโยงอยู่กับ Parent Entity ตัวนั้นๆ เพื่อให้ข้อมูลที่ถูกเรียงลำดับกับข้อมูลจริงที่ปรากฏตรงกันเสมอ

## ⚠️ 4. ข้อควรระวังและแนวทางปฏิบัติที่สำคัญ (Critical Precautions)

> [!CAUTION]
> ### 1. ตั้งค่าความยาว String ของ Sorted IDs Attribute เป็น Unlimited (ไม่จำกัด)
> หากคุณกำหนดความยาว String ของ Sorted IDs ไว้คงที่ (เช่น 200 หรือ 500 ตัวอักษร) เมื่อไอเทมในลิสต์มีจำนวนเพิ่มขึ้น (เช่น มี 20-30 ไอเทมขึ้นไป) ข้อความ GUIDs ที่คั่นด้วยจุลภาคจะมีความยาวเกินลิมิต ส่งผลให้ **Mendix เซฟข้อมูลไม่สำเร็จ (Database Truncation Error)**
> * **แนวทางแก้ไข**: ใน Domain Model ให้ตั้งค่าความยาวของ Attribute ตัวนี้เป็น **Unlimited** (ไม่จำกัด) เสมอ เพื่อรองรับการเซฟลำดับจำนวนมาก

> [!WARNING]
> ### 2. หลีกเลี่ยงการแก้ไขค่าใน Sorted IDs Attribute ด้วยตนเองโดยตรง
> ฟังก์ชันการเรียงลำดับพึ่งพาโครงสร้าง GUID ภายในของระบบ Mendix (เช่น `281474976710657`) ซึ่งไม่ใช่ฟิลด์ที่มนุษย์ทั่วไปอ่านเข้าใจง่าย
> * **แนวทางแก้ไข**: ไม่ควรเขียน Microflow เข้าไปดัดแปลงข้อมูลใน String นี้ด้วยวิธี Manual เช่น ค้นหาหรือแทนที่ String ด้วยตนเอง ปล่อยให้ Widget เป็นตัวจัดการสลับสับค่าเองทางหน้าจอ 100%

> [!NOTE]
> ### 3. การทำความสะอาดขยะ (GUID Garbage Collection) หลังการลบ
> เมื่อมีการลบข้อมูลไอเทมลูก (Child Object) ออกจากระบบ ฐานข้อมูลของ Mendix จะทำความสะอาดข้อมูลตัวตนของมันออกไป แต่ข้อความ GUID ใน Sorted IDs ของ Parent Entity อาจจะยังมีข้อความเลข GUID เดิมคาอยู่
> * **แนวทางปฏิบัติ**: แม้ว่าระบบภายในของ Widget ได้ออกแบบมาให้สแกนข้ามและละเว้น (Ignore) GUID ขยะที่ตรวจไม่พบออบเจกต์ในระบบให้โดยอัตโนมัติแล้วก็ตาม แต่เพื่อความคลีนของฐานข้อมูล คุณสามารถเขียน Microflow ใน `After Delete` event ของตัวลูก เพื่อเข้าไปลบคำสะกดของ GUID ตัวย่อยออกจาก String ของตัวแม่ได้

> [!IMPORTANT]
> ### 4. การจัดการไอเทมที่เพิ่มเข้ามาใหม่ให้เป็นระเบียบ
> หากมีไอเทมใหม่ถูกสร้างขึ้นมาในดาต้าเบส โดยที่ยังไม่เคยมี GUID ของมันเซฟไว้ในตัวแปร Sorted IDs ของ Entity แม่มาก่อน
> * **พฤติกรรมอัตโนมัติของ Widget**: จะจับไอเทมใหม่ปัดสอยไปวางไว้ **ท้ายที่สุดของลิสต์รายการเสมอ** ป้องกันไม่ให้ไอเทมใหม่ตกหล่นจากบอร์ด

## 💡 5. ตัวอย่างกรณีศึกษาในการออกแบบโมเดล (Real-world Examples)

### ตัวอย่างที่ A: ระบบกระดานงานคัมบัง (Kanban Board)

ถ้าคุณกำลังออกแบบกระดานที่มี คอลัมน์ (เช่น To Do, In Progress, Done) และในแต่ละคอลัมน์มีงานที่สามารถสลับลำดับบนล่างได้

```
[ Domain Model ]
┌──────────────────────────────────────┐          ┌──────────────────────────────────────┐
│             KanbanColumn             │          │                 Task                 │
├──────────────────────────────────────┤          ├──────────────────────────────────────┤
│ - ColumnName (String)                │ 1      * │ - TaskTitle (String)                 │
│ - SortedTaskIds (String - Unlimited) ├─────────►│ - Description (String)               │
└──────────────────────────────────────┘          └──────────────────────────────────────┘
```
* **ตัวแปร Context ของหน้าจอ**: ผูกกับ `KanbanColumn`
* **ตัวแปรสลับลำดับ**: ผูก `SortedTaskIds` บนคอลัมน์นั้นๆ
* **เมื่อย้ายงานข้ามคอลัมน์**: 
  1. ลบ GUID ของ Task ออกจากฟิลด์ `SortedTaskIds` คอลัมน์เดิม
  2. สลับ Association ของ Task ตัวนั้นมาผูกกับ คอลัมน์ใหม่
  3. เพิ่ม GUID เข้าไปใน `SortedTaskIds` ของคอลัมน์ใหม่

### ตัวอย่างที่ B: ระบบจัดลำดับคอร์สบทเรียน (Online Course Organizer)

ถ้าต้องการจัดเรียงลำดับหัวข้อย่อยหรือสไลด์บทเรียนในระบบคอร์สการเรียนรู้แบบลากสลับตำแหน่ง

```
[ Domain Model ]
┌──────────────────────────────────────┐          ┌──────────────────────────────────────┐
│             CourseModule             │          │             CourseLesson             │
├──────────────────────────────────────┤          ├──────────────────────────────────────┤
│ - ModuleTitle (String)               │ 1      * │ - LessonTitle (String)               │
│ - SortedLessonIds (String-Unlimited) ├─────────►│ - DurationInMinutes (Integer)        │
└──────────────────────────────────────┘          └──────────────────────────────────────┘
```
* **ตัวแปร Context ของหน้าจอ**: ผูกกับ `CourseModule`
* **ตัวแปรสลับลำดับ**: ผูก `SortedLessonIds`
* **ประโยชน์**: สามารถสร้างลำดับจัดแสดงวิชาเรียนตามใจผู้สอน และบันทึกลงไปใน String แอตทริบิวต์เดียวได้อย่างง่ายดาย โดยไม่ต้องเขียนตรรกะ Order index แบบคำนวณคณิตศาสตร์ตัวเลข (+1 / -1) ทุกครั้งที่มีการสอดตำแหน่งใหม่
