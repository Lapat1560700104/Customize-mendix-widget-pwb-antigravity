# 📑 คู่มือสถาปัตยกรรมข้อมูลและการออกแบบ Entity: SortedTaskIds

เอกสารฉบับนี้จัดทำขึ้นเพื่ออธิบายแนวคิด ที่มา และแนวทางการออกแบบโครงสร้างข้อมูล (Domain Model) ในการใช้งานคุณสมบัติ **`Sorted IDs Attribute`** (หรือที่เรียกทางเทคนิคว่า **`SortedTaskIds`**) ของวิจเจต **`pwbCustomizeContainerDataView`** ให้แก่ผู้ออกแบบระบบและนักพัฒนา Mendix

---

## 🧠 1. แนวคิดและที่มาของระบบ SortedTaskIds

ในระบบลากวางแบบสลับตำแหน่ง (Drag & Drop sorting) ทั่วไป นักพัฒนามักประสบความยากลำบากในการบันทึกลำดับของการ์ด เพราะแนวคิดดั้งเดิมมักใช้แอตทริบิวต์ประเภทตัวเลข เช่น `OrderIndex` (Integer) บันทึกไว้ในการ์ดแต่ละใบ ซึ่งมีข้อเสียร้ายแรงในระดับองค์กรดังนี้:

*   **ปัญหาของระบบดั้งเดิม (`OrderIndex`):**
    1.  **ประสิทธิภาพต่ำ:** เมื่อผู้ใช้แทรกการ์ดไว้ด้านบนสุด ระบบต้องโหลดการ์ดใบอื่นๆ ทั้งหมดขึ้นมา แล้วรันลูปบวกค่าดัชนี (`Index + 1`) ทีละตัวเพื่อเซฟลงฐานข้อมูล ส่งผลให้ฐานข้อมูลทำงานหนักมาก (Write Overhead)
    2.  **ปัญหาค่าซ้ำ (Concurrency Conflict):** หากมีผู้ใช้สองคนใช้งานบอร์ดพร้อมกัน ค่าตัวเลขลำดับอาจซ้ำซ้อนและทำให้ลำดับเพี้ยนได้ง่าย

*   **ทางออกอัจฉริยะด้วย `SortedTaskIds` (String Serialization):**
    วิจเจตนี้จึงแก้ปัญหาด้วยการเปลี่ยนมาเก็บลำดับการ์ดทั้งหมดในรูปแบบ **"ข้อความรหัสไอดีต่อกันคั่นด้วยจุลภาค" (Comma-separated GUIDs String)** ไว้ที่ **Entity พ่อ/แม่ (Parent/Context Entity)** เพียงแอตทริบิวต์เดียว:
    > ค่าที่บันทึก: `"562949953421313,562949953421314,562949953421315"`
    
    *   **ข้อดี:**
        *   **รวดเร็วและเบาบาง:** ทุกจังหวะลากวางสำเร็จ เขียนบันทึกข้อมูลเพียง **1 Record** (บน Entity พ่อ) แทนการเขียนบันทึกนับร้อยไอเทม
        *   **ปราศจาก Concurrency Conflict:** ลำดับจะสอดคล้องกับพฤติกรรมสุดท้ายของผู้ใช้งานรายนั้นๆ เสมอ

## ❓ 2. ความจำเป็นของการเตรียมข้อมูลตั้งต้น (Do we need to pre-create it?)

> **คำถาม:** เราจำเป็นต้องสร้างหรือคำนวณข้อความชุดนี้เตรียมไว้ล่วงหน้าตั้งแต่แรกก่อนเปิดหน้าจอหรือไม่?
> **คำตอบ:** **ไม่จำเป็นเลยค่ะ! ปล่อยให้มีสถานะเป็นค่าว่าง (`Empty` หรือ `Null`) ในฐานข้อมูลได้ 100% ค่ะ**

### กลไกการประมวลผลเมื่อค่าเป็นว่าง (Hydration Logic):
เมื่อผู้ใช้อัปโหลดหน้าเพจขึ้นมาครั้งแรกสุด วิจเจตจะทำงานโดยมีลำดับขั้นตอนดังนี้:
1.  **ดึงข้อมูลลำดับธรรมชาติ:** หากตรวจพบว่าค่าในแอตทริบิวต์เป็นค่าว่าง ตัววิจเจตจะดึงการ์ดทั้งหมดขึ้นมาแสดงผลโดยอ้างอิงตาม **ลำดับการสืบค้นข้อมูลจาก Items Source** ใน Mendix ทันที (เช่น เรียงตามวันเวลาสร้างล่าสุด)
2.  **จังหวะเขียนข้อมูล:** ทันทีที่ผู้ใช้ขยับเมาส์หรือใช้นิ้วลากการ์ดสลับที่บนบราวเซอร์สำเร็จครั้งแรก วิจเจตจะทำการแปลงไอดีการ์ดทุกลำดับในหน้าจอนั้นเป็น String แล้วเขียนเซฟกลับลงไปในแอตทริบิวต์นี้ให้เองโดยอัตโนมัติ

### การจัดการไอเทมใหม่ที่เพิ่งสร้างขึ้นมาภายหลัง:
หากนักพัฒนาเพิ่ม **"การ์ดใบใหม่ (ใบที่ D)"** เข้าไปในฐานข้อมูลหลัก ซึ่งรหัสไอดีของ D ยังไม่อยู่ในข้อความที่เคยเซฟไว้เดิม (`"A,B,C"`)
*   ตัววิจเจตมีตรรกะประมวลผลป้องกันข้อมูลสูญหาย โดยมันจะทำการดึงการ์ด A, B, C มาเรียงลำดับเดิมที่เคยลากไว้ให้ก่อน 
*   จากนั้นจะตรวจพบว่า **การ์ด D เป็นไอเทมใหม่** และนำมันไป **ปรากฏตัวต่อท้ายแถวให้โดยอัตโนมัติ** (เป็นลำดับ: A → B → C → **D**) ทำให้ข้อมูลไม่ตกหล่นและพร้อมให้ผู้ใช้ลากสลับคิวต่อได้ทันที

## 🎨 3. แนวทางการออกแบบโครงสร้างข้อมูล (Entity Design Patterns)

ผู้ออกแบบระบบสามารถนำรูปแบบการสร้าง Entity ใน Domain Model ไปเลือกใช้งานได้ตามความเหมาะสมของหน้าจอ 2 สไตล์ดังนี้ค่ะ:

### 🏆 รูปแบบ A: บอร์ดลิสต์เดี่ยวจัดความสำคัญ (Simple Priority List)
*เหมาะสำหรับ: รายการงานทั่วไป, สินค้าขายดี, การจัดคิวจัดความสำคัญของใบสั่งงาน (Ticketing)*

```
┌────────────────────────────────────┐
│              Project               │ (Entity พ่อ/แม่หลักประจำหน้าจอ)
├────────────────────────────────────┤
│ - Name (String)                    │
│ - SortedTaskIds (String - Unlimited) ◄─── [ช่องผูก Sorted IDs Attribute]
└────────────────────────────────────┘
                  │ 1
                  │ (ความสัมพันธ์ 1-to-Many: Project_Task)
                  │ *
┌────────────────────────────────────┐
│               Task                 │ (Entity ลูกที่เป็นแผ่นการ์ดแต่ละใบ)
├────────────────────────────────────┤
│ - Title (String)                   │
│ - Description (String)             │
└────────────────────────────────────┘
```

---

### 🏆 รูปแบบ B: บอร์ดคัมบังลากสลับคอลัมน์ (Multi-column Kanban Board)
*เหมาะสำหรับ: บอร์ดจัดการงานสไตล์ Trello (ลากข้ามระหว่างคอลัมน์ และขยับสลับลำดับการ์ดขึ้นลงในแต่ละตู้แยกจากกัน)*

```
       ┌────────────────────────┐
       │         Board          │ (Entity บอร์ดหลักสูงสุด)
       ├────────────────────────┤
       │ - BoardName (String)   │
       └────────────────────────┘
                   │ 1
                   │
         ┌─────────┴─────────┐
         │ *                 │ * (ความสัมพันธ์ร่วมบอร์ด)
         ▼                   ▼
┌─────────────────┐ 1     * ┌─────────────────┐
│   BoardColumn   │◄────────┤      Task       │ (Entity ลูกการ์ดงาน)
├─────────────────┤Task_Col ├─────────────────┤
│ - Caption (Str) │         │ - Title (Str)   │
│ - StatusVal(Str)│         │ - Status (Enum) ├── [ช่องผูก Item Column Attribute]
│ - SortedTaskIds ◄─────────│                 │    (ใช้อัปเดตสถานะการ์ดอัตโนมัติ)
└─────────────────┘         └─────────────────┘
  (String - Unlimited)        (เก็บสถานะปัจจุบันของการ์ด
  [ช่องผูก Sorted IDs]         เช่น 'ToDo', 'Done')
  (เก็บลำดับการ์ดแยกตู้)
```

#### ⚙️ การตั้งค่าหน้าจอสำหรับรูปแบบคัมบัง (Kanban Settings):
เมื่อวางคอลัมน์ของบอร์ด ให้คุณใช้ **Mendix Data View ของ `BoardColumn`** ครอบวิจเจตนี้ไว้ในแต่ละตู้ และกรอกค่าพารามิเตอร์ดังนี้:
1.  **Items Source (DataSource):** ดึงข้อมูล `Task` โดยใช้ XPath คัดกรองการ์ดเฉพาะตู้ตัวเอง:
    `[Task_Board = $currentBoard and Status = $currentColumn/StatusVal]`
2.  **Sorted IDs Attribute:** เลือกผูกกับตัวแปร **`BoardColumn.SortedTaskIds`** เพื่อให้แต่ละคอลัมน์เซฟลำดับภายในตู้ของตัวเองแยกอิสระ
3.  **Kanban Support:** ตั้งค่าเป็น `True`
4.  **Drag Group Name:** ตั้งค่าชื่อเดียวกันทุกตู้ เช่น `'project-kanban'`
5.  **Column Value:** เลือกผูกกับ **`BoardColumn.StatusVal`**
6.  **Item Column Attribute:** เลือกผูกกับ **`Task.Status`** เพื่อสั่งการให้เปลี่ยนสถานะของการ์ดเป็นตู้ใหม่ทันทีเมื่อหล่นสัมผัสพื้น

## 📝 4. สรุปขั้นตอนปฏิบัติเพื่อประสิทธิภาพสูงสุด
1.  **ตั้งขนาด String เป็น Unlimited:** แอตทริบิวต์ที่มาผูกกับ `Sorted IDs Attribute` ควรมีขนาดเป็น Unlimited หรือขนาดใหญ่เพียงพอ เพื่อป้องกันปัญหาสายอักขระขาดเมื่อมีจำนวนไอดีการ์ดมากขึ้น
2.  **การลบข้อมูล (Garbage Collection):** เมื่อผู้ใช้ลบการ์ดลูกออกจากฐานข้อมูล วิจเจตจะทำการปัดรหัสไอดีที่ไม่มีอยู่จริงนั้นออกจากสายอักขระให้เองในการเรนเดอร์ครั้งถัดไป แต่เพื่อความสะอาดของข้อมูล คุณสามารถเขียนคำสั่งใน Microflow ลบการ์ดให้ไปนำไอดีนั้นออกจาก `SortedTaskIds` บนตัวพ่อได้เช่นกันค่ะ
